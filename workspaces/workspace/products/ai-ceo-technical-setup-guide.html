2. OpenClaw Installation

2.1 Quick Installation (Recommended)

1

Download OpenClaw

# Install OpenClaw globally via npm npm install -g openclaw # Verify installation openclaw --version
2

Initialize Workspace

# Create workspace directory mkdir -p ~/.openclaw/workspace cd ~/.openclaw/workspace # Initialize configuration openclaw init
3

Configure API Keys

# Create secrets file (NEVER commit this) cat > ~/.openclaw/secrets.json << 'EOF' { "openai_api_key": "sk-your-key-here", "anthropic_api_key": "sk-ant-your-key-here", "google_ai_key": "your-key-here" } EOF # Set permissions (read/write only by owner) chmod 600 ~/.openclaw/secrets.json
⚠️ Security Warning: The secrets.json file contains sensitive API keys. Never share this file or commit it to version control.

2.2 Manual Installation (Advanced)

1

Clone from Source

# Clone repository git clone https://github.com/openclaw/openclaw.git cd openclaw # Install dependencies npm install # Build from source npm run build # Link globally npm link

2.3 Docker Installation

1

Run with Docker

# Pull latest image docker pull openclaw/openclaw:latest # Run container docker run -d \ --name openclaw \ -v ~/.openclaw:/root/.openclaw \ -p 18789:18789 \ openclaw/openclaw:latest
✅ Installation Complete! You should now be able to run openclaw status to verify your installation.

3.2 Deploying Your First Agent

1

Create Agent Configuration

# Create agents directory mkdir -p ~/.openclaw/workspace/agents # Create agent config cat > ~/.openclaw/workspace/agents/content-agent.yaml << 'EOF' name: content-agent type: content model: openai/gpt-4 schedule: "0 6,12,18 * * *" # 6am, 12pm, 6pm daily tasks: - name: "daily_content" description: "Generate daily content schedule" template: "content/daily-schedule" - name: "social_posts" description: "Create social media posts" template: "content/social-posts" EOF
2

Activate Agent

# Deploy agent openclaw agent deploy content-agent.yaml # Verify agent status openclaw agent status content-agent # View agent logs openclaw agent logs content-agent --tail 50

4. Automation Workflows

4.1 Workflow Design Principles

Before creating workflows, understand these core principles:

📘 Key Principles:
  • Start Simple: Begin with 2-3 steps, add complexity later
  • Test Each Step: Verify individually before chaining
  • Add Error Handling: Always include failure notifications
  • Document Everything: Future you will thank present you

4.2 Workflow Template Structure

# Workflow Template name: "workflow-name" description: "What this workflow does" trigger: type: "schedule|event|webhook" config: schedule: "0 9 * * *" # Daily at 9am conditions: all: - field: "status" equals: "approved" actions: - step: "step-1" action: "http-request" config: url: "https://api.example.com" method: "POST" - step: "step-2" action: "send-email" config: to: "user@example.com" template: "welcome" error_handling: - on_failure: action: "notify" config: channel: "slack" message: "Workflow failed: {error}"

4.3 Common Workflow Patterns

Pattern 1: Lead Capture to CRM

# Lead Capture Workflow name: "lead-to-crm" trigger: type: "webhook" source: "form-submission" actions: - action: "validate-email" - action: "add-to-crm" config: system: "airtable|hubspot|salesforce" - action: "send-welcome-email" config: template: "welcome-sequence" - action: "notify-team" config: channel: "slack"

Pattern 2: Content Distribution

# Content Distribution Workflow name: "content-distribution" trigger: type: "schedule" config: schedule: "0 9 * * 1-5" # Weekdays 9am actions: - action: "fetch-content" config: source: "content-calendar" - action: "generate-social-posts" config: platforms: ["twitter", "linkedin", "facebook"] - action: "schedule-posts" config: tool: "buffer|hootsuite|later" - action: "log-distribution"

Pattern 3: Revenue Tracking

# Revenue Tracking Workflow name: "revenue-tracking" trigger: type: "webhook" source: "stripe-payment" actions: - action: "parse-payment" - action: "update-dashboard" config: system: "google-sheets|airtable|notion" - action: "check-milestone" config: threshold: [1000, 5000, 10000] - action: "notify-milestone" config: channel: "discord|slack|telegram"

4.4 Workflow Testing Checklist

  • Submit test data through trigger
  • Verify each action executes correctly
  • Check data maps to correct fields
  • Test edge cases (empty fields, special characters)
  • Verify error handling works
  • Test with real data before production

Configure Stripe in OpenClaw

# Add Stripe credentials cat > ~/.openclaw/config/integrations/stripe.yaml << 'EOF' provider: stripe api_key: ${STRIPE_SECRET_KEY} webhook_secret: ${STRIPE_WEBHOOK_SECRET} mode: live # or 'test' for sandbox products: - id: prod_xxx name: "AI CEO Toolkit" price: 299700 # $2,997 in cents currency: usd webhooks: - event: "checkout.session.completed" action: "fulfill-order" - event: "payment_intent.succeeded" action: "update-revenue" EOF

Test Stripe Integration

# Test with Stripe CLI stripe login stripe listen --forward-to localhost:18789/webhooks/stripe # Test webhook locally stripe trigger checkout.session.completed

5.2 Messaging Integration (Discord)

1

Create Discord Bot

# Add Discord credentials cat > ~/.openclaw/config/integrations/discord.yaml << 'EOF' provider: discord bot_token: ${DISCORD_BOT_TOKEN} client_id: ${DISCORD_CLIENT_ID} guild_id: ${DISCORD_GUILD_ID} channels: notifications: "123456789" # Channel ID for notifications revenue: "123456790" # Channel ID for revenue updates alerts: "123456791" # Channel ID for system alerts permissions: - send_messages - embed_links - read_message_history EOF

6.2 Automated Health Monitoring

1

Create Health Check Script

#!/bin/bash # health-check.sh - Run every hour via cron openclaw status > /var/log/openclaw/health.log 2>&1 if [ $? -ne 0 ]; then echo "OpenClaw health check failed at $(date)" >> /var/log/openclaw/errors.log openclaw restart fi
2

Add to Crontab

# Edit crontab crontab -e # Add health check (every hour) 0 * * * * /home/user/.openclaw/scripts/health-check.sh

6.3 Log Management

# View logs in real-time openclaw logs --follow # Export logs for analysis openclaw logs --export logs-$(date +%Y%m%d).json # Search logs for errors openclaw logs --filter "error" --tail 100 # Archive old logs (run weekly) find ~/.openclaw/logs -name "*.log" -mtime +7 -exec gzip {} \;

6.4 Backup Procedures

#!/bin/bash # backup.sh - Run daily BACKUP_DIR="/backup/openclaw/$(date +%Y%m%d)" mkdir -p $BACKUP_DIR # Backup configuration cp -r ~/.openclaw/config $BACKUP_DIR/ # Backup workspace cp -r ~/.openclaw/workspace $BACKUP_DIR/ # Backup logs (last 7 days) find ~/.openclaw/logs -name "*.log" -mtime -7 -exec cp {} $BACKUP_DIR/ \; echo "Backup completed: $BACKUP_DIR"

7. Security Configuration

7.1 API Key Security

⚠️ Critical Security Rules:
  • Never commit API keys to version control
  • Use environment variables for secrets
  • Rotate keys every 90 days
  • Use separate keys for development/production
# Set environment variables (add to ~/.bashrc or ~/.zshrc) export OPENAI_API_KEY="sk-your-key-here" export ANTHROPIC_API_KEY="sk-ant-your-key-here" export STRIPE_SECRET_KEY="sk_live_your-key-here" # Load environment variables source ~/.bashrc

7.2 Network Security

1

Configure Firewall

# Allow only localhost access (default) sudo ufw allow from 127.0.0.1 to any port 18789 # Or allow specific IP ranges sudo ufw allow from 192.168.1.0/24 to any port 18789 # Enable firewall sudo ufw enable

7.3 SSL/TLS Configuration

1

Generate SSL Certificate

# Install certbot sudo apt install certbot # Generate certificate sudo certbot certonly --standalone -d yourdomain.com # Certificate location # /etc/letsencrypt/live/yourdomain.com/fullchain.pem # /etc/letsencrypt/live/yourdomain.com/privkey.pem
2

Configure HTTPS in OpenClaw

# Update OpenClaw config cat > ~/.openclaw/config/server.yaml << 'EOF' server: host: 0.0.0.0 port: 18789 ssl: enabled: true cert: /etc/letsencrypt/live/yourdomain.com/fullchain.pem key: /etc/letsencrypt/live/yourdomain.com/privkey.pem EOF

8. Troubleshooting Guide

8.1 Common Issues & Solutions

Issue Cause Solution
Agent won't start Missing API key Check environment variables
Workflow fails silently No error handling Add error notifications
API rate limited Too many requests Add delays, use batching
Webhook not receiving Wrong endpoint URL Verify webhook configuration
Memory errors Context too large Clear memory, optimize prompts
Slow response Network/model latency Use faster model or cache
